# 前端用户体验审计

日期：2026-06-05

## 审计范围

- 目标：检查现有竞品分析工作台是否支持用户快速创建分析、理解运行状态、浏览证据并获取可用报告。
- 流程：首次进入 -> 去配置 -> 新建配置 -> 加载历史分析 -> Agent 追踪 -> 最终报告 -> 移动端查看。
- 方式：在本地运行 FastAPI 与 React/Vite 前端，使用浏览器实际操作与 DOM/响应式检查。
- 限制：本轮未提交新的分析任务，避免向历史任务列表写入额外数据；未进行完整屏幕阅读器测试。

## 逐步健康度

1. 首次进入：一般。主 CTA 清楚，但首屏同时展示大标题、零值指标、11 步流程和长侧边栏，用户不容易判断先看什么。
2. 去配置：良好。空状态提供了明确的“去配置”动作，符合任务入口预期。
3. 新建配置：一般。表单基本可用，但缺少模板来源提示、字段示例的中文化和字段级错误反馈。
4. 加载历史分析：较差。加载后保留原滚动位置，用户可能直接落在结果页中段；相同产品名的历史记录难以区分。
5. Agent 追踪：较差。默认首先展示大量底层事件，业务用户很难快速找到失败、工单和关键结论。
6. 最终报告：一般。结构化报告与功能证据地图有价值，但页面重复展示结构化内容和 Markdown 全文，且中英文混杂。
7. 移动端查看：阻断。侧边栏在主内容之前完整纵向展开，用户需要滚动数千像素才能看到报告。

## 优先级整改清单

### P0：修复移动端主内容被长侧边栏阻断

问题：小于 980px 时，双栏直接变为单列，侧边栏仍完整排在工作区之前。历史记录和 provider 说明让用户必须滚动很长距离才能到主内容。

如何修改：

- 移动端将侧边栏改为顶部紧凑导航栏或抽屉。
- 默认只显示品牌、当前任务和“菜单”按钮；模板、历史分析、模型模式放入抽屉。
- 选择模板或历史分析后自动关闭抽屉并将焦点移动到主内容标题。
- 增加“跳到主内容”链接。

验收：390x844 视口进入页面后，首屏能看到主任务标题和主要操作；打开历史结果后无需滚动侧边栏即可看到结果摘要。

### P1：把结果页默认入口从底层追踪改成业务摘要

问题：加载历史结果后默认进入 Agent 追踪，大量事件、英文事件类型和 provider 元数据淹没关键结论。业务用户需要自己寻找最终报告或证据风险。

如何修改：

- 新增“概览”作为默认标签页，展示核心结论、差异、机会点、风险、待复核工单和报告入口。
- Agent 追踪放到最后一个“运行详情”标签，默认折叠正常事件，只突出失败、fallback、工单和异常。
- 在顶部增加明确主动作：“查看报告”“查看待复核项”“继续新分析”。
- 加载历史分析或切换标签时重置主工作区滚动位置。

验收：用户打开历史分析后，10 秒内能回答“结论是什么、可信度如何、下一步做什么”。

### P2：重构信息架构，减少首屏和结果页的重复信息

问题：大标题、零值指标、11 步流程、可信度摘要、任务配置、模板规则和报告章节同时抢占注意力。最终报告又同时展示结构化模块和完整 Markdown，内容重复。

如何修改：

- 首页只保留价值说明、主 CTA、最近一次分析和两个模板入口；零值指标和流程步骤在运行后再展示。
- 结果页顶部只保留一组摘要：可信度、关键风险、任务范围、更新时间。
- 将“报告章节 / 结论类型 / 证据规则 / 复核门禁”折叠到“分析配置详情”。
- 最终报告默认展示结构化版本；Markdown 全文放入“原始报告”折叠区或单独标签。
- 报告章节条改成可点击目录，并显示当前阅读位置。

验收：桌面端结果首屏只出现一个明确摘要区和一组主操作，不重复展示相同指标或章节。

### P3：提升配置表单的可理解性与错误恢复

问题：“查询竞品目标”默认值和“目标读者”仍为英文；错误只在表单底部展示第一条，开始分析按钮禁用时不解释原因；模板卡点击会直接覆盖当前编辑内容。

如何修改：

- 将默认分析目标改为自然中文，或用可选择的分析维度复选项生成目标。
- 将目标读者改为中文预设下拉，并允许自定义。
- 每个错误显示在对应字段旁，按钮禁用时提供原因提示。
- 模板卡使用“使用此模板”明确动作；存在未保存编辑时，切换模板前确认。
- 增加“保存为配置模板”和“重置”动作。

验收：用户不看说明文档也能完成配置；故意清空必填项后可立即知道哪里错、如何修。

### P4：统一中文、术语和状态表达

问题：界面仍混有 Demo、Agent、mock、fixture、provider、fallback、positioning、ai_capability 等内部术语；报告正文大量中英文混排，降低阅读速度。

如何修改：

- 建立统一术语表，界面优先显示中文，英文技术名放在提示或详情中。
- 将 `mock / fixture` 显示为“演示数据”，将 provider 显示为“数据提供方式”。
- 对来源原文保留英文，但给结论、事件摘要和报告结构化字段提供中文摘要。
- 修复重复标题，例如“定价模型 定价模型”“用户画像 用户画像”。

验收：非研发用户可以在不理解 provider、fixture、claim 等词的情况下完成主任务。

### P5：补齐无障碍和视觉细节

问题：缺少统一的 `:focus-visible` 样式；刷新图标按钮仅 28x28px；标签页没有 tab 语义；多个状态主要依赖颜色和浅色徽标；密集小字影响可读性。

如何修改：

- 为按钮、链接、输入框、选择框增加高对比度 `:focus-visible` 焦点环。
- 交互目标最小提升到 44x44px。
- 标签页使用 `role="tablist"`、`role="tab"`、`aria-selected` 和对应面板关系。
- 状态同时使用图标、文本和颜色；检查文本/背景对比度。
- 为动态运行状态、错误和成功消息增加 `aria-live`。
- 将 11 步进度在移动端改为当前步骤 + 已完成数量，而不是完整纵向列表。

验收：仅用键盘可完成模板选择、配置、标签切换和报告浏览；200% 缩放无内容丢失。

## 已有优势

- 空状态“去配置”提供了明确入口。
- 配置已从侧边栏移入主工作区，任务创建与导航职责更清楚。
- 证据与结论页面支持搜索、筛选、排序和展开证据。
- 功能证据地图支持搜索和展开，具备进一步产品化的基础。
- 可信度摘要、证据覆盖率和复核工单形成了较强的信任表达。

## 建议实施顺序

1. P0 移动端导航与主内容可达性。
2. P1 概览页、默认入口与滚动重置。
3. P2 信息架构去重。
4. P3 配置表单反馈。
5. P4 术语中文化。
6. P5 无障碍与视觉细节。
